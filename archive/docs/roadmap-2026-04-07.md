# Meta-Harness Roadmap

更新时间：2026-04-08

## 1. 目标

把 `meta-harness` 从当前的离线实验编排内核，推进为一个：

- 通用的 eval control plane
- 可产品化部署的评测平台
- 带真实外部集成的 observability / dataset / benchmark / optimization 系统

目标状态分三层：

1. 内核层：稳定的数据模型、artifact 规范、dataset/evaluator/benchmark 能力
2. 产品层：API、权限、多用户、UI、任务调度、质量门
3. 生态层：OTel、Phoenix、Langfuse、对象存储、外部 agent / app 适配器

## 2. 当前基线

截至 2026-04-08，仓库已经具备：

- CLI 驱动的 `candidate -> run -> score -> observe -> benchmark -> propose -> shadow-run` 闭环
- run/candidate artifact 落盘
- task-set 执行、workspace 物化、patch candidate、command evaluator
- benchmark suite、stability/cost ranking、strategy card 基础能力
- failure signature 检索、trace exporter skeleton、failure dataset 提取
- 统一 `run_search_loop()`、proposal ranking / evaluation、task plugin 与 integration outer-loop 收口
- FastAPI surface、service layer、inline async jobs 与基础 bearer token 校验
- dataset lifecycle、gate policy / promotion artifact、white-box audit、runtime profiling
- per-evaluator trace artifact 与 profiling 摘要
- 命名集成目录、带 timeout/retry/failure classification 的 HTTP export path 与 OTLP / Phoenix / Langfuse vendor payload export
- CI smoke/regression/nightly benchmark workflows

当前不具备或尚未产品化的部分：

- 完整多用户 API 产品面与 workspace 权限模型
- 数据库索引层与查询模型
- 更稳的 OTel / Phoenix / Langfuse transport 与对象存储 / 远程 artifact 同步
- 更完整的 evaluator tracing / profiling 治理
- 更完整的 dataset / annotation 运营流转
- UI 产品面
- job queue / worker 的独立运行时
- gate policy 与 champion promotion 的审批 / 运营化收口

## 3. 路线原则

- 文件系统仍是事实源，数据库只做索引、查询、聚合、权限与加速
- 先稳定 canonical schema，再做服务化和 UI
- 先接真实外部 observability，再扩展产品面
- 执行、评分、观测、优化保持解耦
- 所有自动决策都必须有 artifact、trace、score 和 lineage 可回放

## 4. 阶段规划

## 阶段状态快照

- Phase 0：已完成。`ADR`、`data model`、`api surface`、`artifact contract`、`gate policy` 已在文档层落地。
- Phase 1：基本完成。`TraceEventV2`、`EvaluatorRun`、dataset lifecycle、compaction / archive / retention、最小 evaluator tracing / profiling 已具备；剩余缺口主要是更细的 lineage 治理。
- Phase 2：部分完成。最小 HTTP export path、vendor payload export 与 integration health check 已落地，但 transport hardening、对象存储、远程 artifact 同步仍未完成。
- Phase 3：部分完成。API / service surface、inline async jobs 与基础 token 校验已存在，但 DB projection、workspace 权限、独立 worker 和完整产品化查询模型仍未完成。
- Phase 4 以后：尚未启动产品级落地，仍以路线图形式保留。

## Phase 0：架构定稿与边界冻结

建议周期：2026-04-07 到 2026-04-18

目标：

- 冻结 v1 平台边界
- 定稿 canonical data model
- 定义产品化最小闭环

交付物：

- `docs/adr/` 目录与首批 ADR
- `docs/data-model-v1.md`
- `docs/api-surface-v1.md`
- `docs/artifact-contracts.md`
- `docs/gate-policy-v1.md`

重点事项：

- 明确 `Run`、`TaskExecution`、`TraceEventV2`、`EvaluatorRun`、`DatasetVersion`、`BenchmarkExperiment`、`BenchmarkVariant`
- 统一 artifact 命名、目录规范、版本字段
- 定义“什么写文件，什么进索引库，什么暴露 API”
- 明确支持的适配对象：CLI agent、HTTP app、RAG system、code repo workflow

验收标准：

- schema 结构在文档中稳定，不再边做边改
- 所有现有 CLI 命令都能映射到目标对象模型
- 对外集成边界和非目标项写清楚

## Phase 1：内核产品化

建议周期：2026-04-21 到 2026-05-23

目标：

- 把当前内核从“可用”推进到“可稳定复用”

工作流：

- Trace 模型升级
- Evaluator 体系升级
- Dataset 体系升级
- Artifact 生命周期升级

交付物：

- `TraceEventV2` 最终落地
- `EvaluatorRun` artifact 目录结构
- `datasets/` 规范与 dataset manifest
- task-set 到 dataset 的正式兼容层
- run compaction / archive / retention policy v1

重点事项：

- 给 trace 增加 `session_ref`、`artifact_refs`、`prompt_ref`、`retrieval_refs` 的稳定语义
- evaluator 运行结果单独建模，支持重评分和 evaluator artifact 保存
- dataset 支持 `dataset_id`、`version`、`tags`、`split`、`source_type`、`expectations`
- 增加 dataset freeze 和 benchmark 输入快照

验收标准：

- 相同 run 可多次重评分且结果可审计
- dataset 可从 task set、failed runs、手工样本三种来源统一表达
- benchmark 可引用固定 dataset version
- artifact 保留/压缩/归档策略可配置

## Phase 2：真实外部集成

建议周期：2026-05-26 到 2026-06-27

目标：

- 从 adapter skeleton 升级到真实外部系统接入

工作流：

- OTel export / transport
- Phoenix 集成
- Langfuse 集成
- 对象存储与远程 artifact 同步

交付物：

- OTLP exporter
- Phoenix exporter / ingestion path
- Langfuse exporter / ingestion path
- S3 兼容对象存储 artifact backend
- `configs/integrations/*.json`

重点事项：

- 支持本地文件和远程 transport 双写
- 保留 run/task/evaluator/candidate 上下文映射
- 增加 integration health check
- 为外部平台生成稳定 trace/session/observation id

当前状态：

- 已有最小 OTLP / Phoenix / Langfuse HTTP export path、基础 timeout/retry/failure classification 与 vendor payload export。
- 仍缺批量、重试、协议校准、远程失败治理、对象存储 backend。

验收标准：

- 单个 run 能真实发送到 OTLP endpoint
- Phoenix / Langfuse 至少能看到 run 级 trace 和 task spans
- 远程失败不影响本地 canonical artifact 落盘
- 外部集成开关支持 project/profile 级配置

## Phase 3：服务化与多用户

建议周期：2026-06-30 到 2026-08-01

目标：

- 从纯 CLI 工具升级到可被产品和团队消费的平台服务

工作流：

- API 层
- 索引数据库
- 身份权限
- 异步任务调度

交付物：

- `server/` 或 `src/meta_harness/api/` 服务入口
- PostgreSQL 或 SQLite + migration 方案
- 用户、workspace、project、token 数据模型
- job queue / worker 模型
- OpenAPI 文档

建议首批 API：

- `POST /runs`
- `GET /runs`
- `GET /runs/{run_id}`
- `POST /runs/{run_id}/score`
- `POST /benchmarks`
- `GET /benchmarks/{experiment_id}`
- `POST /datasets`
- `GET /datasets/{dataset_id}/versions/{version}`
- `POST /annotations`
- `POST /candidates/{candidate_id}/promote`

重点事项：

- 数据库只存索引和查询投影，不替代本地 artifact
- API 与 CLI 共享 service 层，避免双份逻辑
- 引入后台任务状态：`queued/running/succeeded/failed/cancelled`
- 明确权限边界：谁能看 run，谁能 promote champion，谁能改 gate policy

当前状态：

- 基础 API、service split 与 inline job facade 已存在。
- 仍缺 workspace 权限、独立 queue / worker、DB migration 与更完整查询模型。

验收标准：

- UI 和外部系统能只通过 API 使用平台
- 单个 benchmark suite 可异步执行和查询进度
- CLI 命令底层改为调用统一 service 层
- 至少支持单机多用户和 token 认证

## Phase 4：产品面落地

建议周期：2026-08-04 到 2026-09-12

目标：

- 做出第一版真正可用的 eval 平台产品界面

首批页面：

- Runs
- Run Detail
- Trace Explorer
- Benchmarks
- Datasets
- Candidates / Champions
- Integrations
- Gate Policies

交付物：

- 前端应用
- run 列表与过滤器
- benchmark 对照视图
- dataset 浏览与版本选择
- candidate/champion 管理页
- integration status 页

重点事项：

- 所有视图都要围绕 run/candidate/benchmark/dataset 真实对象建模
- 先做检索、过滤、diff、趋势，不追求复杂可视化
- trace 视图最少要支持 task/phase/status/error/artifact drill-down
- benchmark 视图要能看 baseline、best variant、ranking penalty、stability

验收标准：

- 非开发者可通过 UI 完成 run 查看、benchmark 对比、candidate promote
- 至少支持最近一次 run 的 trace 和 score 追溯
- 产品闭环不再依赖直接读文件目录

## Phase 5：质量门与运营化

- 把平台真正接入日常工程流程

工作流：

- PR gate
- nightly benchmark
- 回归集冻结
- champion promotion 工作流
- alerting / reporting

交付物：

- gate policy engine
- benchmark regression report
- champion approval workflow
- Slack / webhook 通知

重点事项：

验收标准：

- PR 可以被平台结果阻断或放行
- nightly suite 有固定输出和失败告警
- candidate/champion 变更都有 lineage 和 evidence

## 6. 优先级

P0：

- schema 定稿
- evaluator tracing
- dataset 版本化
- OTLP 真接入
- API/service 抽象

P1：

- Phoenix / Langfuse 真接入
- gate policy engine
- benchmark suite 产品化
- candidate/champion 工作流
- UI 首版

P2：

- annotation queue
- 多租户权限深化
- 对象存储归档
- adapter 扩展
- 生态模板与跨项目能力

## 7. 里程碑定义

M1：内核稳定可复用

- schema 冻结
- evaluator/dataset/trace 三件套到位
- 所有现有 CLI 命令迁移到统一 service 逻辑

M2：真实外部可观测性接通

- OTLP、Phoenix、Langfuse 至少两条真实可用
- 外部故障不破坏本地 canonical artifacts

M3：平台服务化

- API 可驱动 runs、benchmarks、datasets、candidates
- 支持鉴权、异步任务、索引查询

M4：产品首发

- UI 可完成日常浏览、对比、追溯、promotion
- gate policy 能接 CI

M5：通用平台化

- 至少 3 类外部系统完成接入
- 新项目接入成本降到配置优先

## 8. 风险

- schema 未冻结就先做 API，会导致频繁破坏兼容
- 把数据库做成事实源，会破坏当前 artifact-first 优势
- UI 先行会导致服务层和对象模型反复返工
- 外部平台深度耦合会让内部模型失去自主性
- benchmark 与 gate policy 不分离，会把研究实验和生产阻断混成一套

## 9. 成功标准

当满足以下条件时，可认为项目已经从“离线实验工具”升级为“产品化 eval 平台”：

- 平台可同时服务多个项目和多个用户
- 运行、评分、对比、追溯、promotion 可通过 API 和 UI 完成
- trace、dataset、benchmark、candidate 都有稳定对象模型与版本语义
- Phoenix / Langfuse / OTel 至少两类真实外部集成可用
- CI gate 与 nightly benchmark 成为常规工程流程的一部分
- 新项目接入主要通过配置和 adapter，而不是复制改代码

## 10. 下一步建议

立刻开始的顺序：

1. 补 evaluator tracing / profiling，并把 trace lineage、多维评分和自动 shadow validation 收口。
2. 把 OTLP / Phoenix / Langfuse / 对象存储从最小 export path 推进到更稳的 transport 与远程 artifact 方案。
3. 为 API / async job 增加更明确的 workspace 权限、独立 worker 与 DB projection 路线。
4. 在服务层稳定后再推进 UI、审批流与运营化工作流。
